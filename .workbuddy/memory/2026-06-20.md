## 20:41 - curl 链接到 Homebrew openssl 问题排查

**初步修复**：curl.sh 添加 `export PKG_CONFIG_LIBDIR="$PKG_CONFIG_PATH"`，将 pkg-config 锁定在 EServer Library 内。但 otool 显示 libcurl.dylib 仍依赖 `/opt/homebrew/opt/openssl@3/lib/libssl.3.dylib`。

## 22:09 - 找到 curl 链接 Homebrew openssl 的真正根因

**问题**：curl 自己的 configure 正确找到了 EServer 的 openssl@3.5（config.log 证实：
`pkg-config: SSL_LDFLAGS: "-L/Applications/EServer/Library/openssl@3.5/lib"`），
但 otool 显示 libcurl.dylib 仍依赖 `/opt/homebrew/opt/openssl@3/lib/libssl.3.dylib`。

**根因**：传递依赖问题。
- `libngtcp2.sh` 第 12 行 `OPENSSL_PREFIX=/Applications/EServer/Library/openssl`（路径错误，实际是 `openssl@3.5`）
- 由于路径不存在，pkg-config fall back 到系统路径，找到 Homebrew 的 openssl@3
- libngtcp2_crypto_ossl.dylib 被链接到 Homebrew openssl
- curl 链接 libngtcp2_crypto_ossl 时，传递依赖被记录到 libcurl.dylib 的 Mach-O load commands

**修复**：
- `libngtcp2.sh`：OPENSSL_PREFIX 改为 `openssl@3.5`，并加 PKG_CONFIG_LIBDIR
- `libssh2.sh`：加 PKG_CONFIG_LIBDIR（保险加固）

**关键经验**：
1. otool 看到的依赖可能不是直接依赖，而是传递依赖
2. PKG_CONFIG_PATH 只是在默认路径前追加搜索，PKG_CONFIG_LIBDIR 才是完整锁定搜索范围
3. macOS dylib 的 install_name 会自动传递给依赖它的二进制
